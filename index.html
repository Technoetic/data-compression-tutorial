§1 RLE — Run-Length Encoding (런-길이 부호화)

연속된 같은 값을 "값 × 횟수"로 줄이는 가장 단순한 무손실 압축. 종이의 90%가 흰색인 팩스에서는 효율적이지만, 모든 글자가 다른 텍스트에는 오히려 2배가 됩니다.

  • 팩스 (ITU-T T.45)
  • BMP / TGA / PCX
  • JPEG 양자화 후처리
  • Windows 3.x 부팅 화면

§2 허프만 부호화 — Huffman Coding

자주 나오는 문자에 짧은 비트열을, 드문 문자에 긴 비트열을 주는 가변 길이 부호화. 1952년 MIT 박사과정생 데이비드 허프만이 제안. 접두사 무관(prefix-free) 트리로 디코딩이 유일합니다.

  • DEFLATE 2단계 (ZIP / PNG / gzip)
  • JPEG 엔트로피 코딩
  • MP3 인코딩
  • .docx / .xlsx 내부 압축

§3 LZ77 — 사전식 슬라이딩 윈도우 압축

"방금 본 패턴이 또 나오면, 그 시작 거리·길이를 적어 두기." Lempel과 Ziv가 1977년에 제안. PNG·ZIP·gzip의 핵심 1단계입니다. (거리 D, 길이 L, 다음 글자 c) 토큰을 출력합니다.

  • DEFLATE 1단계 (ZIP / PNG / gzip / zlib)
  • GIF (LZW 변형)
  • 7-Zip (LZMA)
  • Brotli (HTTP/Web)

§4 Base64 — 부호화(인코딩)지, 압축이 아닙니다

바이너리 데이터를 ASCII 64글자(A-Z, a-z, 0-9, +, /)로 표현. 텍스트 전용 채널(이메일 SMTP·URL·JSON)에 바이너리를 실어 보낼 때 씁니다. 크기는 오히려 약 33% 증가합니다.

  • 이메일 첨부 (SMTP 7-bit)
  • data: URI (HTML/CSS 이미지)
  • JWT 토큰 (Base64URL)
  • QR 코드 / PGP 지문

§5 종합 — PNG · ZIP · HTTP gzip은 어떻게 만들어질까

DEFLATE는 LZ77 + Huffman을 합친 알고리즘입니다. ZIP·PNG·gzip·.docx·.xlsx 모두 내부에서 이 두 단계를 차례로 사용합니다. Base64는 이 과정과 무관하게, 압축된 바이너리를 텍스트 채널로 보낼 때 별도로 씁니다.

1단계 · LZ77

원본 텍스트의 반복을 (거리, 길이, 다음 글자) 토큰으로 치환합니다.

2단계 · Huffman

토큰과 리터럴을 빈도에 따라 가변 길이 비트열로 부호화합니다.

필요하면 결과 바이트를 Base64로 감싸 텍스트 채널에 실어 보냅니다(예: 이메일에 .zip 첨부).
